Managing namespaces with ArgoCD in multi-tenant clusters

Create and configure namespaces, quotas and limit ranges using ArgoCD on Openshift Gitops
February 8, 2022

Introduction

Cluster Administrators need to create namespaces for their developer teams and also provision these namespaces with required Resource Quotas and Limit Ranges. The developer teams own these namespaces and are admins to its contents but they are not allowed to modify the quotas and limits.

This blog is intended to document the ways in which the creation, configurations and management of multiple namespaces can be simplified by using ArgoCD.

Resource Quotas

Resource Quota is a Kubernetes object that allows users to control or limit the CPU or memory consumed within a namespace. These quotas can also be used to limit the numbers of resources that can be created in a namespace.

Limit Ranges

Resource Quota is a Kubernetes object that allows users to control or limit the CPU or memory consumed within a namespace. These quotas can also be used to limit the numbers of resources that can be created in a namespace.

Creation of a Namespace Management Application using ArgoCD

ArgoCD is a powerful continuous delivery tool for Kubernetes resources. Namespaces and quotas are Kubernetes resources, hence we can leverage ArgoCD to create and manage them. ArgoCD also supports using Kustomize and Helm Charts for creating applications.

This section will first document the steps for creating a simple ArgoCD application from a Git Repository. Following that, 2 more better alternative ways of creating the application using Kustomize and Helm will be discussed.

1. Simple ArgoCD Application using Git Repository

  1. Create a Git Repository and save the config files for the namespaces, quotas, limit ranges. For Example -
    • namespace.yaml
      • apiversion: v1
      kind: Namespace
      metadata:
        name: dev
        annotations:
          argocd.argoproj.io/sync-wave: "-1"
        label:
          argocd.argoproj.io/managed-by: openshift-gitops
    • resource-quota.yaml
      • apiversion: v1
      kind: ResourceQuota
      metadata:
        namespace: dev
        annotations:
          argocd.argoproj.io/sync-wave: "0"
      spec:
        hard:
          pods: "10"
    • limit-range.yaml
      • apiversion: v1
      kind: LimitRange
      metadata:
        namespace: dev
        annotations:
          argocd.argoproj.io/sync-wave: "0"
      spec:
        limits:
        - default:
            cpu: 200m
            memory: 512Mi
          defaultRequest:
            cpu: 100m
            memory: 256Mi
          type: Container
  2. Adding sync waves in these yaml files is important, because Namespace needs to be created before the quotas and limit ranges can be applied to them by argocd controller
  3. Login to your Openshift cluster as administrator
  4. The argoCD application controller needs additional permissions to create Resource Quotas and Limit Ranges in the cluster namespaces. Openshift Cluster Roles and Role Bindings need to be used to grant additional permissions to application controller of the default argocd instance in the openshift-gitops namespace to allow creation of Resource Quotas and Limit Ranges

    • In order to this, we need to create additional cluster-roles and cluster-rolebindings and apply to the cluster

    • oc create -f cluster-role.yaml
      • apiVersion: rbac.authorization.k8s.io/v1
      kind: ClusterRole
      metadata:
        name: quota-limits-cluster-role
      rules:
      - apigroups: [""]
        resources: ["resourcequotas", "limitranges"]
        verbs: ["create"]

    • oc create -f cluster-role-binding.yaml
      • apiVersion: rbac.authorization.k8s.io/v1
      kind: ClusterRoleBinding
      metadata:
        name: quota-limits-cluster-role-binding
      roleRef:
        apiGroup: rbac.authorization.k8s.io
        kind: ClusterRole
        name: quota-limits-cluster-role
      subjects:
      - kind: ServiceAccount
        name: openshift-gitops-argocd-application-controller
        namespace: openshift-gitops
  5. Create an Argocd application with this Git Repository, refresh and synchronize it. This will now create the namespace and related quotas and limit ranges.

Limitation

Keeping individual manifests for each team will make managing these config files a tedious task, especially when there are a large number of teams and projects. Hence, better approaches that make use of template tools like Kustomize and Helm are discussed in the following sections.

2. Use Kustomize to create and manage resources

  1. By using custom patches in Kustomize we can avoid creating multiple manifests and reuse common elements from resource quotas and limit ranges. See example repository - https://github.com/saumeya/blog-example-repo/tree/main/kustomize-namespace-config/teams
  2. The teams folder defines all the patches for different teams onto the base manifests. On creating the argoCD application all these resources are created.
  3. Follow the last three steps (3, 4 and 5) from the previous section to grant additional permissions and create ArgoCD applications to manage your namespaces.

3. Use Helm and Application Set to create and manage resources

Helm templates can also be used to parametrize values for namespace, resource quotas and limit ranges configurations. This will however, require you to create more ArgoCD applications to manage these namespaces. To avoid creating the multiple applications one by one, Application Sets can be leveraged to specify the value files and create all the applications in one go.

  1. Create a git repository and add the following resource files, under Templates folder in the helm folder structure - create namespace.yaml, limit-range.yaml and resource-quota.yaml
    • namespace.yaml
      • apiversion: v1kind: Namespace
      metadata:
        name: {{ .Values.namespace }}
        labels:
          name: {{ .Values.namespace }}
        annotations:
          argocd.argoproj.io/sync-wave: "-1"
    • resource-quota.yaml
      • apiversion: v1
      kind: ResourceQuota
      metadata:
        namespace: {{ .Values.namespace }}
        annotations:
          argocd.argoproj.io/sync-wave: "0"
      spec:
        hard:
          pods: {{ .Values.quota.pods }}
    • limit-range.yaml
      • apiversion: v1
      kind: LimitRange
      metadata:
        namespace: {{ .Values.namespace }}
        annotations:
          argocd.argoproj.io/sync-wave: "0"
      spec:
        limits:
        - default:
            cpu: {{ .Values.limits.default.cpu }}
            memory: {{ .Values.limits.default.memory }}
          defaultRequest:
            cpu: {{ .Values.limits.defaultRequest.cpu }}
            memory: {{ .Values.limits.defaultRequest.memory }}
          type: {{ .Values.limits.type }}

      These values are default values and the admins can configure their own values using custom value files.

    • values.yaml
      • namespace: default-dev

      limits:
        default:
        cpu: 200m
        memory: 512Mi
        defaultRequest:
          cpu: 100m
          memory: 256Mi
        type: Container

      quota:
        pods: "10"
  2. Application Sets can be used to create multiple applications for namespace creation for different fields. Check the example repo for the teams folder and add relative path of the custom value files in the list generator, this will create the namespace app for specific teams. Apply the following Application Set config to your cluster.
  3. Follow steps 4 and 5 from the previous approach to grant additional permissions on your cluster and create an argocd application.
  4. Follow the last three steps (3, 4 and 5) from the first approach to grant additional permissions and create ArgoCD applications to manage your namespaces. More information about how to create applications using Helm can be found here.

Conclusion

This blog documents the ways in which ArgoCD can be used to simplify the process of creating, managing and configuring namespaces. Anytime a change is required in the quotas or limits, the admins need to simply modify the config files and ArgoCD will do the rest of the work for them.
